# 《不许跳！》制作总纲

> 别名：制造总纲。  
> 用途：把分散在策划、美术、程序、测试文档里的大框架收拢成团队每日对齐用的总地图。  
> 原则：本文件只写方向、边界和制作顺序；具体数值、坐标、文案原句仍以对应专项文档为准。

## 1. 一句话定位

《不许跳！》是一款 2D 横版反冲攀登游戏：玩家没有普通跳跃键，只能通过枪械后坐力把自己崩上去，在实验室里完成一系列荒诞但规则清晰的测试。

核心卖点：

- 没有跳跃键。
- 枪口指向哪里，玩家就往反方向飞。
- 失败后快速重试，鼓励玩家不断调整角度和落点。
- 普通路线宽容，捷径和成绩挑战承担高难度。

## 2. 当前锁定目标

短期目标不是把完整游戏一次做完，而是先把“手枪章节”做成可靠样板。

当前优先级：

1. 保持 D1 已跑通的核心闭环稳定。
2. 收敛 0-1、0-2、0-3、D1 的手枪教学流程。
3. 确保新玩家能在 60 秒内理解“不能跳，但能用后坐力飞”。
4. 补足最低成本的手感反馈：开火、飞行、落地、重生。
5. 为美术接入预留结构，但不因正式资源未到而阻塞关卡和手感验证。

## 3. 核心体验边界

必须保留：

- 无普通跳跃键。
- 反冲移动结果必须稳定、可学习、无随机冲量。
- 掉落后快速重生，计时不清零。
- R 键完全重开，计时、事故计数、位置全部清零。
- 终点到达后冻结玩法并展示成绩。
- PC 与手机共用物理参数，设备差异只放在输入适配层。

不能引入：

- 战斗和敌人 AI。
- 实时多人。
- 程序生成地图。
- 养成、商店、货币系统。
- 复杂剧情分支、长日志、长演出。
- 外部运行时、原生库、本地服务或自由外网依赖。

## 4. 主流程结构

主流程按“实验训练 + 毕业测试”组织。

| 区域 | 主要职责 | 当前制作状态 |
|---|---|---|
| 入职校准 | 教会反冲方向、斜向转向、空中补枪救险 | 当前重点 |
| 精准弹射科 | 强化手枪精准落点、连续平台、窄平台 | 下一阶段 |
| 大力出奇迹科 | 引入霰弹枪，大冲量、大缺口、捷径 | 手枪稳定后再做 |
| 空中姿态科 | 引入三连发，空中多段调整和曲线飞行 | 中后期 |
| 毕业测试 | 串联多武器，形成最终综合关 | 后期 |

降级原则：

- 如果时间不足，火箭筒可以降级为毕业关的一次性发射台。
- 如果手枪手感没过，不继续扩新武器。
- 如果手机适配明显拖慢进度，先保证横屏核心体验，手机只做可玩验证。

## 5. 武器定位

四种武器必须各自服务不同关卡问题，不能互相成为完全上位替代。

| 武器 | 玩家感受 | 主要用途 | 预览/辅助方向 |
|---|---|---|---|
| 校准手枪 | 精准、标准、可控 | 普通路线、窄平台、落点练习 | 地面显示完整轨迹和落点 |
| 霰弹枪 | 大力、笨重、风险高 | 大缺口、捷径、夸张喜剧效果 | 显示半段轨迹 |
| 三连发 | 柔性、可调整、空中修正 | 曲线飞行、绕障碍、移动平台 | 空中允许调整方向 |
| 火箭筒 | 高风险、高收益、仪式感 | 毕业测试、高级捷径 | 只显示初始方向或蓄力提示 |

当前只实现和验证手枪。其他武器先保留配置结构，不急于进入完整开发。

## 6. 关卡设计原则

每个房间只回答一个问题：“玩家在这里要学会或证明什么？”

通用原则：

- 新机制第一次出现必须安全。
- 第二次要求应用。
- 第三次才允许明显损失。
- 普通路线宽容，捷径承担精度和速度压力。
- 空中补枪不能成为所有玩家的强制门槛，应优先作为救险、提速或高分手段。
- 玩家连续失败时，应能看懂自己错在角度、时机或落点，而不是觉得系统随机。

手枪章节当前目标：

| 关卡 | 教学目标 | 关键约束 |
|---|---|---|
| 0-1 | 枪口向下，角色向上 | 平台宽，允许慢慢理解 |
| 0-2 | 斜向开火改变飞行方向 | 不依赖空中补枪 |
| 0-3 | 空中补枪可以救险 | 普通路线不强制补枪 |
| D1 | 综合校准 | 验证前 3 项能力组合 |

## 7. 美术方向

已确认方向：

- 整体为纯白、干净、整洁的实验室。
- 游戏默认横屏，只做横屏优先。
- 玩家角色为小巧的非人形吉祥物方向，当前反馈偏向小巧兽人形象。
- 第一屏就要让玩家理解“禁止跳跃，但允许用枪移动”的荒诞规则。
- 平台应逐步从纯矩形替换为实验室内可能存在的物件。
- 终点是发光的实验室门或手术室门。
- 危险区可以红色半透明，但后续需要警示线或图案，不能只靠颜色区分。

美术第一批优先级：

1. 角色基础动作：待机、瞄准、开火后坐、起飞、空中飞行、落地、失败重生、通关胜利。
2. 手枪章节场景件：普通平台、窄平台、终点门、危险区警示。
3. 手感特效：枪口闪光、飞行拖尾、落地飞尘。
4. HUD 图标：时间、当前武器、空中补枪、危险区、重试。

待确认：

- 正式素材导出格式。
- 基准尺寸。
- 资产命名规则。
- 枪械是否单独绘制并旋转，或使用左右水平翻转。

## 8. 文案与叙事语气

已确认方向：

- 玩家是第 404 号实习测试员。
- 场景是“稳稳当当装备测试中心”。
- 语气轻松荒诞，不做黑暗反转。
- 笑点落在规则矛盾、角色反应和失败场面，而不是嘲讽玩家。
- 不做对话树、长演出、长日志和复杂人物关系。

文案使用方式：

- 开场不超过 60 字。
- 每区广播 3-4 条，每条约 3 秒读完。
- D1 只需要短句，不做完整剧情演出。
- 失败吐槽准备约 12 条，避免连续重复。
- 每件武器需要 1 句功能说明和 1 句笑话。
- 结算保留普通、优秀、极限成绩 3 种短变体。

当前待确认：

- 正式中文名是否锁定《不许跳！》。
- 是否需要正式英文名。
- 区域命名最终润色。
- 毕业测试“三分钟准时下班”和门禁卡笑点是否最终保留。
- 发布简介、截图文案和 TapTap 活动材料负责人。

## 9. 程序制作顺序

当前代码已经完成 D1 核心闭环，并扩展到入职 3 房、UI 页面流转、轨迹预览、开火闪光和飞行拖尾。

接下来程序顺序：

1. 测试当前 0-1、0-2、0-3、D1 是否都能稳定通关。
2. 调整 0-3，让空中补枪成为救险/捷径，而不是普通路线强制门槛。
3. 增加空中补枪可用/不可用的明显反馈。
4. 增加落地反馈和掉落重生反馈。
5. 准备美术接入字段：玩家渲染尺寸与碰撞体分离，平台和终点增加 visualType。
6. 手枪章节稳定后，再开发霰弹枪最小版本。
7. 霰弹枪通过后，再评估三连发和火箭筒是否进入本次 MVP。

程序不得猜测 TapTap Code 未确认 API。平台能力必须从当前文档或编辑器确认后再接入。

## 10. 测试门槛

手感验证优先于内容扩展。

核心通过线：

- 80% 玩家能在 60 秒内无需口头解释完成第一次目标落地。
- 至少 60% 玩家失败后 3 秒内主动重试。
- 玩家能把失败归因于角度、时机或落点，而不是游戏随机。
- 掉落到恢复操作不超过 1.5 秒。
- 新玩家主流程目标为 15-25 分钟。
- 发布前从启动到开始游戏不超过 3 次输入。

如果指标不过，只调整输入、冲量、镜头、平台、反馈和提示，不新增武器或新系统。

## 11. 当前最重要的制作判断

现在项目不是缺“大方向”，而是需要把大方向压成可执行的小任务。

最近一阶段只追一个问题：

> 手枪章节能不能让一个没听过规则的人，在 1 分钟内理解“不能跳，但可以用后坐力飞”，并愿意再试一次？

只要这个成立，后续武器、关卡、美术、文案都有可靠地基。  
如果这个不成立，继续扩内容只会扩大返工。
