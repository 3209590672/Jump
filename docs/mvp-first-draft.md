# 《不许跳！》（暂定名）

> TapTap「制造新星 Game Jam」MVP 第一版策划案  
> 主题：跳跃｜类型：2D 横版物理平台动作｜平台：PC + 手机｜团队：程序、美术、文案各 1 人｜周期：20 天

## 0. 一页结论

- **一句话卖点**：没有跳跃键；瞄准地面开枪，用后坐力把自己一路崩上实验塔。
- **玩家幻想**：从乱飞的测试菜鸟，成长为能连续精准弹射的实验室毕业生。
- **情绪目标**：荒诞、紧张、侥幸翻盘、逐渐熟练；不追求羞辱式挫败。
- **完整体验**：10–15 分钟主流程，首次游玩约 20–40 分钟；通关后保留成绩重试和本地最佳成绩。
- **传播画面**：一枪飞出屏幕、空中补救、擦边落地、高手一炮跨越半个房间。
- **MVP 成败判据**：陌生玩家 60 秒内理解规则，3 分钟内产生“再来一次”。

[→ game-design-fundamentals：30 秒核心循环与克制设计；Miyamoto：手感优先]

## 1. 核心循环

```text
观察落点 → 选择安全路线或捷径 → 瞄准并开火
    ↑                              ↓
学习与进度 ← 落地或坠落 ← 反冲飞行与空中补救
```

30 秒内至少发生一次“预判—开火—失控—补救—落地”。选择来自起跳位置、开火角度、力度以及稳妥路线与捷径的取舍。[→ game-design-fundamentals：有意义的决策；Sid Meier：游戏是一连串有趣的选择]

### MDA 与目标玩家

- **机制**：反向冲量、有限空中弹药、四种枪械曲线、平台机关。
- **动态**：惯性叠加、临场救险、安全路线与速通路线分化。
- **体验**：挑战、感官刺激、幽默、轻叙事。
- **主要玩家**：成就型；次要为探索型与速通竞争者。

[→ game-design-fundamentals：MDA 与玩家动机；Raph Koster：掌握过程本身就是乐趣]

## 3. 核心机制

### 基础运动

- 地面可慢速左右移动；垂直跨越只能依靠后坐力。
- 枪口方向决定相反方向的反冲；同样输入必须得到同样结果，禁止随机冲量。
- 落地补满当前武器弹药；空中弹药有限，形成补救时机的选择。
- 坠出房间或碰到危险区后快速重生，最长 1.5 秒恢复操作。
- 摄像机提前展示主要落点，不要求向屏幕外盲跳。

[→ game-design-fundamentals：技巧可学习；Jonathan Blow：机制诚实]

### PC 与手机输入

| 行为 | PC | 手机 |
|---|---|---|
| 左右移动 | A/D 或方向键 | 左侧水平摇杆 |
| 瞄准 | 鼠标按住拖动 | 右侧拖动 |
| 开火 | 松开按键 | 松手 |
| 取消 | 右键/Esc | 拖回取消圈 |

- 两端统一为“按住瞄准—松开开火”。
- 显示方向线和力度刻度，不显示完整预测轨迹。
- 手机只提供轻微角度吸附；若发布后接入平台榜单，再考虑按设备类型筛选。
- 不做持续射击武器，避免触屏失控。

[→ game-design-fundamentals：输入反馈一致；Miyamoto：关卡直接教学]

### 四种武器

| 武器 | 规则 | 作用 |
|---|---|---|
| 校准手枪 | 中等固定冲量，空中 1 发 | 精准、标准路线 |
| 蹦蹦霰弹枪 | 大冲量，空中 1 发，冲量无随机 | 大缺口、捷径、笑点 |
| 三连点射器 | 固定方向分 3 次小冲量 | 曲线飞行、柔性落地 |
| 蓄力火箭筒 | 按住分三级力度，空中不可再开火 | 毕业测试、高手捷径 |

每件武器只改变一个主要变量，避免同时学习力度、弹量、延迟和散布。[→ game-design-fundamentals：少量规则产生组合；Richard Garfield：优雅系统]

### 成败与计分

- 主流程：完成三段实验训练并通过毕业测试。
- 成绩记录排序：是否完成 → 净用时 → 死亡次数。
- 首发只保留本地最佳成绩，不做每日挑战和日榜。
- 主流程有区域检查点；重试时保留快速回到当前练习内容的节奏。

[→ game-design-fundamentals：失败促进学习；Jenova Chen：保持心流]

## 4. 内容与难度

| 区域 | 首次时长 | 教学职责 | 强度 |
|---|---:|---|---:|
| 入职校准 | 1–2 分钟 | 反向冲量、首次落地、空中补救 | 2/10 |
| 精准弹射科 | 2–3 分钟 | 手枪、角度、连续落点 | 4/10 |
| 大力出奇迹科 | 3–4 分钟 | 霰弹、大跨越、捷径 | 6/10 |
| 空中姿态科 | 3–4 分钟 | 三连发、移动平台、曲线飞行 | 7/10 |
| 毕业测试 | 2–3 分钟 | 四武器串联、限时压力 | 8/10 |
| 结算小剧场 | 30–60 秒 | 成绩、笑点、再次挑战入口 | 1/10 |

每区采用“安全介绍—组合发展—变化—休息笑点”。难度是坡升、平台、峰值、释放，不是直线。[→ level-design：三拍教学与节奏谷值；Jenova Chen：心流]

### 毕业测试：《三分钟准时下班》

广播宣布下班，出口三分钟后关闭。玩家依次完成手枪精准登阶、霰弹跨越、三连发绕挡板，最后取得火箭筒。结尾同时提供三段稳妥路线和“一炮到出口”的高手捷径。

抵达后才发现门禁卡落在起点；广播沉默两秒，旁边员工通道自动打开。笑点不改变成绩，也不否定玩家努力。[→ level-design：高潮后释放；Miyamoto：最终关检验已学机制]

详细灰盒见 [level-outline.md](./level-outline.md)。

## 5. 世界与文案

玩家是“第 404 号实习测试员”，参加一家不太靠谱的枪械实验室毕业考核。实验室认为跳跃会导致安全事故，因此禁止跳跃，却允许员工用大威力武器移动。

剧情重要度 3/5：服务笑点、节奏和角色记忆，不做沉重内核。叙事通过实验广播、区域标题、失败短句、环境标语和结算小剧场完成；不做对话树和长演出。[→ game-design-fundamentals：叙事服务循环；Fumito Ueda：克制]

### 文案预算

- 开场不超过 60 字。
- 每区广播 3–5 条，每条约 3 秒读完。
- 失败吐槽 12 条，避免连续重复。
- 每件武器 1 句功能说明＋1 句笑话。
- 正常通关与优秀成绩各 1 个短结局变体。
- 成绩结算与再次挑战按钮文案。

详见 [world-lore.md](./world-lore.md)。

## 6. UI、反馈与可访问性

### 必要界面

- 标题页：开始实验、设置、制作人员。
- HUD：计时、当前武器、空中弹药、当前区域。
- 瞄准：方向线、力度刻度、取消圈。
- 失败：短冻结、轨迹残影、吐槽字幕、快速重试。
- 结算：完成时间、死亡次数、最佳成绩、再次挑战。

### 手感最低标准

- 开火：枪口闪光、角色挤压拉伸、2–4 帧停顿、短屏震、清晰低频音效。
- 飞行：速度拖尾和镜头前置。
- 擦边落地：独立音效、小粒子、角色晃动。
- 失败：重试时清空输入，避免误触再次开火。

[→ game-design-fundamentals：动作—反馈闭环；Miyamoto：手感优先]

### 可访问性

- 字幕默认开；关键音频有视觉替代。
- 平台以轮廓和图案区分，不只依赖颜色。
- 可调屏震、残影、瞄准吸附、文本大小。
- 练习模式允许房间中继点，但不计入本地最佳成绩。
- PC 支持键位调整；手机适配安全区和屏幕比例。

[→ level-design：可访问性基线；Jenova Chen：不同能力玩家均可进入心流]

## 7. 音频

- AI 生成 1 条可循环实验室主题曲和 1 个毕业测试加速变体。
- 约 12–16 个 SFX：4 种开火、落地/擦边/撞击、危险区、检查点、倒计时、结算。
- 广播优先用文字＋合成提示音；如做配音，只用一名“过度热情的广播员”。
- 保存工具、提示词、日期和授权条款记录，发布前确认活动与商用合规。

## 8. TapTap Code 技术边界

> 以下以团队提供的平台限制为前提；实际 API 名称必须在 TapTap Code 当前文档或编辑器核验，不在策划中猜测。

- 客户端 Lua 实现玩法，逻辑放在 `scripts/`，资源放在 `assets/`。
- 不依赖外部运行时、原生库、本地服务或自由外网。
- 本地处理单机物理；首发只做本地成绩，云变量、存档或平台榜单发布后再评估。
- 物理参数数据化，PC 与手机读取同一套数据。
- 不做每日挑战，也不生成随机地图。

```text
scripts/
├─ config/weapon_config.lua, level_config.lua
├─ core/game_state.lua, event_bus.lua
├─ gameplay/player_controller.lua, recoil_system.lua
│  weapon_system.lua, checkpoint_system.lua
├─ ui/hud_controller.lua, aim_controller.lua, mobile_controls.lua
└─ platform/save_adapter.lua, leaderboard_adapter.lua
assets/
├─ art/
├─ audio/
├─ levels/
└─ ui/
```

存档只需：最高区域、最佳时间、死亡数、设置。若后续接平台能力，只记录主流程成绩所需的时间、死亡数、设备和关卡版本。平台 API 未确认前只定义适配层，不硬编码接口。[→ game-design-fundamentals：数据驱动与序列化；Richard Garfield：保持最小]

## 9. 20 天计划

| 天数 | 主要成果 | 验收 |
|---|---|---|
| D1–D2 | 单房间反冲、落地、重试 | 空房间弹射也有趣 |
| D3 | PC/手机输入原型 | 60 秒内无口头指导完成首次跳跃 |
| D4–D6 | 相机、检查点、手枪与霰弹 | 两把枪差异明显 |
| D7–D9 | 四武器、机关、全流程灰盒 | 从头到尾可通关 |
| D10–D12 | 美术、UI、文案接入 | 10–15 分钟流程成立 |
| D13–D14 | 毕业测试、结算、音频 | 可录完整宣传视频 |
| D15 | 本地最佳成绩、发布材料整理 | 不做每日挑战和日榜 |
| D16–D17 | 性能、适配、Bug 修复 | PC/手机各完整通关 3 次 |
| D18 | 5 人外部静默测试 | 至少 3 人愿意再试 |
| D19 | 发布构建、图标、截图、简介 | 审核材料齐全 |
| D20 | 只修阻断问题并发布 | 不新增功能 |

D3 是生死线；若核心手感不过关，停止堆内容，只调冲量、输入、镜头和反馈。[→ game-design-fundamentals：先验证核心循环；Miyamoto：手感优先]

## 10. 明确不做

实时多人、战斗与敌人 AI、程序生成地图、装备养成、货币商店、复杂分支剧情、大型 Boss、外部服务器、原生插件、持续射击武器。

这些功能均不服务 20 天内验证“枪械后坐力跳跃”这一核心卖点。[→ game-design-fundamentals：控制范围；Fumito Ueda：做减法]

## 11. 风险与降级

| 风险 | 处理 |
|---|---|
| 后坐力不爽 | D3 前集中调整冲量、停顿、音效、镜头 |
| 手机明显更难 | 加宽普通平台、轻微吸附，不改物理 |
| 物理不可预测 | 固定步长、速度上限、确定性参数、无随机散布 |
| 玩家不愿重试 | 房间级重生、缩短回退、增加练习模式 |
| 四武器来不及 | 火箭筒改成毕业关的一次性发射台 |
| 榜单接口阻塞 | 首发只保留本地最佳成绩 |
| 美术风格未定 | 先锁轮廓、危险物与落脚面的可读性规范 |
| AI 音频授权不清 | 改用明确许可素材并保留来源清单 |
